# Registry Standard

Conventions for first-class resource registries in a foundation-based project. The foundation supports two patterns: Catalog Resources for explicit content packs, and directory-scanned autoload registries for large generated/global datasets.

## Choose the pattern

Create a registry when you have designer-authored `Resource` data and callers need a single source of truth for "does this X exist" or "what are all the Xs". Skip registries for:

- Runtime-only types (no `.tres` backing).
- Resources embedded inside other resources.
- Small closed sets that belong in constants or enums.

Use a **Catalog Resource** when the content is an authored pack: reward pools, loot tables, decks, wave lists, tutorial bundles, or any content set where order, curated membership, scene injection, test variants, or alternate debug pools matter.

Use a **directory autoload registry** when the content is broad global lookup data: many generated resources of one type, O(1) by-id access across the project, and no authored ordering beyond the resources themselves.

Do not hide content authoring inside a generator/service. Generators and gameplay services may read registries, but the registry or catalog owns the authored content list.

## Catalog Resource

A Catalog Resource is a `.tres` asset whose script extends `Resource` and exports typed arrays or fields pointing at authored content. It owns no runtime state, RNG, save data, or gameplay flow. It is passed into the system that needs it, typically through a scene export or a setup/build method.

| Thing                 | Location                                                           |
| --------------------- | ------------------------------------------------------------------ |
| Catalog script        | `data/<domain>/definitions/<domain>_catalog.gd` or similar         |
| Authored content      | `data/<domain>/...`                                                |
| Catalog asset         | `data/<domain>/<catalog_name>.tres`                                |
| Production wiring     | Scene export, setup argument, or another explicit dependency owner |

Required API:

- `get_all_<plural>() -> Array[T]` or a domain-specific getter that returns a duplicate.
- `get_by_id(id: StringName) -> T` when by-id lookup is needed.
- `validate() -> bool` for null entries, empty ids, duplicate ids, and domain-specific authored-data problems.

Catalog Resources are the preferred pattern for small/medium curated packs and for feature-local content that tests should replace with synthetic data.

## Directory Autoload Registry

A directory autoload registry scans a folder and builds a by-id dictionary at startup. Use it for broad global data, especially generated data.

| Thing                 | Location                                                              |
| --------------------- | --------------------------------------------------------------------- |
| Script                | `global/autoloads/registries/<singular>_registry.gd`                  |
| Resource files        | `data/<domain>/tres/<plural>/` or another documented generated folder |
| Path constant         | `DataPaths.<PLURAL>_DIR`                                              |
| Autoload registration | `project.godot`                                                       |
| Lifecycle             | load resources in `_ready()` via base class                           |

If the `.tres` directory is generated by tooling and gitignored, say so in a comment next to the `DataPaths` constant — a fresh checkout without the generator run will hit an empty registry and fail at boot.

## Base class

All registries extend `ResourceRegistry` (`global/autoloads/registries/resource_registry.gd`). Override two methods:

```gdscript
extends ResourceRegistry

func _dir_path() -> String:
    return DataPaths.<PLURAL>_DIR

func _id_of(r: Resource) -> String:
    return (r as <X>Data).<x>_id if r is <X>Data else ""
```

The base class handles `_ready()` loading and default `validate()` (non-empty check). Override `validate()` to add cross-reference checks.

`validate()` checks **authored content only** — that designer `.tres` files are well-formed and reference ids that exist. It **must not read live/save state**; validating restored save state (live ids still resolve) is the save provider's job, in its `from_dict` / `validate` (see the Save System section of `CLAUDE.md`).

## Required API

Every directory autoload registry exposes, for its resource type `T`:

- `get_<singular>_by_id(id: String) -> T` — O(1) lookup. Returns null on miss.
- `get_all_<plural>() -> Array[T]` — typed wrapper over `get_all()`. Primary iteration API.
- `size() -> int` — inherited from base class; used by `validate()`.

`get_all()` on the base class returns untyped `Array`. The typed `get_all_<plural>()` wrapper on the concrete class is what callers should use. Catalog Resources should expose the same spirit of API, but they do not need to extend `ResourceRegistry` or become autoloads.

**Iterate resources, not ids.** The `_ids` form exists for save/load boundaries, not as the default iteration path.

**Pass Resource refs at non-serialization call sites.** String ids belong at serialization boundaries; refs belong in game logic.

## Forbidden APIs

- **No display-name wrappers.** `get_<singular>_display_name(id)` is banned. Callers take the resource and read `.display_name`.
- **No fallback-to-id accessors.** Any helper of the form "return field X, or the id if missing" hides data bugs.
- **No cross-registry scanning.** Scanning Registry A to answer Registry B's question is the anti-pattern this standard exists to prevent.

## Inverse lookups

When registry A needs to answer "which Bs belong to this A?", prefer in order:

1. **Direct reference on the resource.** If `CategoryData` holds `super_category: SuperCategoryData`, answer via `cat.super_category`.
2. **Index built in `_ready`.** If grouping is needed, build a `Dictionary` at startup after the dependency registry has loaded.
3. **Never by scanning a third registry.**

If your `_ready` builds an index by walking another registry, add `assert(OtherRegistry.size() > 0, "...")` at the top and verify autoload order in `project.godot`.

## Cross-registry method naming

When one registry exposes a lookup keyed by **another** registry's id, use the `_for` suffix:

```
CategoryRegistry.get_super_category_for(category_id) -> SuperCategoryData
```

Rule: if a method's parameter id-type differs from the registry's own id-type, the name ends in `_for`.

## Checklist for a new registry

1. Decide whether the data is a Catalog Resource or a directory autoload registry. Prefer a Catalog Resource for curated packs and explicit injection; use a directory autoload registry for broad generated/global lookup data.
2. For a Catalog Resource: script under `data/<domain>/definitions/`, asset under `data/<domain>/`, exported typed content fields, `get_by_id()` only when useful, and `validate()`.
3. For a directory autoload registry: script at `global/autoloads/registries/<singular>_registry.gd`, extending `ResourceRegistry`.
4. `DataPaths.<PLURAL>_DIR` constant for directory autoload registries. Comment if gitignored.
5. Autoload line in `project.godot`, before `SaveManager`. Check load order if `_ready` reads other registries.
6. Override `_dir_path()` and `_id_of()`. Override `validate()` for cross-reference checks.
7. Required API only: `get_<singular>_by_id`, `get_all_<plural>`, `size`. No display-name wrappers, no fallback accessors.
